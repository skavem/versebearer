# frontend

## Purpose
Operator UI. SvelteKit + Svelte 5 (runes) + Tailwind + DaisyUI. Static-adapter build emits to `dist/`, which the Go binary embeds via `//go:embed all:frontend/dist` and serves inside the Wails webview. Three tabs: Библия (Bible navigation), Песни (song picker + couplet display), Экраны (toggle projector windows).

## For AI Agents

### Working In This Directory
- Pure SPA: `+layout.ts` sets `prerender = true` and `ssr = false`. There is no server-side runtime — every page is static + JS at runtime.
- Wails bindings live at `src/lib/bindings/changeme/...` (autogenerated by `wails3 generate bindings`). Never hand-edit them; rerun `task common:generate:bindings` from the repo root after changing Go method signatures.
- When `wails3 dev` runs, the Vite dev server is launched on port 9245 by default (`VITE_PORT` in root `Taskfile.yml`) and Wails proxies it. The built `dist/` is only used in production builds.
- Svelte 5 syntax: `$state`/`$derived`/`$props`/`$bindable`/`$effect`. Don't downgrade to Svelte 4 patterns (`writable`, `$:` reactivity).
- Tailwind classes use array syntax (`class={["a", cond && "b"]}`) widely — that's Svelte 5's class merging via `tailwind-merge`-compatible behavior.

### Testing Requirements
- Type check: `npm run check` (svelte-kit sync + svelte-check).
- Manual UI check via `wails3 dev` from repo root.

### Common Patterns
- Stores are factory functions (`createXxxStore()`) returning a getter/setter object — not Svelte stores. State is rune-based; cross-component sharing is by exporting the instance.
- Hot-key handling lives inside `+page.svelte`-equivalent route components (`Bible.svelte`, `Songs.svelte`) via `$effect` + `document.addEventListener`. Arrow keys + Enter + Escape are mapped per tab.
